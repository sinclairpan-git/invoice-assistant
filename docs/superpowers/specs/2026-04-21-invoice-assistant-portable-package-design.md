# 发票助手 Windows 便携服务包设计

日期：2026-04-21

## 1. 背景

当前仓库已经完成发票整理助手 `001-009` 的功能收口，适合开发与本地验收，但还不是面向非技术用户的交付形态。目标用户是一位 Windows 助理，需要以尽量低门槛的方式启动、配置、上传发票、复核和导出，而不需要理解 Python、Node、端口、命令行或开发环境。

因此，这一轮不新增产品业务能力，而是把现有系统包装成一个可移植、可交付、可被技术小白使用的本地 Web 服务包，并配套一份极易理解的中文用户指引。

## 2. 目标

本轮目标：

1. 交付一个适用于 Windows 的便携服务包。
2. 用户解压后可通过双击脚本完成启动和停止。
3. 运行时不要求用户自行安装 Python、Node 或前端开发工具。
4. 前端以静态文件方式交付，由后端统一托管，正式交付时只运行一个本地服务进程。
5. 用户第一次使用时，必须先完成“公司发票信息与规则配置”，再进入上传与处理流程；未完成配置前不得开始批次上传。
6. 提供一份面向技术小白的中文 `用户指引.html`，以真实业务场景逐步引导使用。
7. 提供一份面向维护者的技术说明文档，覆盖目录结构、日志、端口、数据迁移和升级方式。

## 3. 非目标

本轮不做以下事项：

1. 不把当前系统改造成真正的原生桌面应用。
2. 不引入新的登录、权限、审批或同步机制。
3. 不改变现有业务规则、OCR 能力、导出契约或审计逻辑。
4. 不做完整安装向导、注册表写入、系统服务注册或 MSI/EXE 安装器。
5. 不新增云部署、公网托管或多人协作能力。

## 4. 备选方案

### 方案 A：开发工程直出

保留当前前后端双进程开发形态，让用户在目标机器上安装 Python、Node 和依赖，再通过脚本启动。

优点：

- 改动最小。
- 与当前仓库运行方式最一致。

缺点：

- 对助理不友好。
- 运行环境不可控，失败点过多。
- 交付效果更像“开发项目”，不是“可用工具”。

### 方案 B：Windows 便携服务包

将前端预构建为静态资源，由后端统一托管；目录内自带运行时和启动脚本，用户解压后双击即可使用。

优点：

- 最接近“一键启动”的目标。
- 只需要维护一个本地服务进程，稳定性更高。
- 迁移和备份简单，适合助理场景。

缺点：

- 包体积较大。
- 需要额外整理打包脚本与目录结构。

### 方案 C：真正安装版桌面软件

进一步封装为带安装向导的 Windows 软件。

优点：

- 用户感知最接近传统软件。

缺点：

- 当前仓库不具备该形态基础。
- 打包和维护成本显著高于用户当前需求。

### 推荐结论

采用方案 B：Windows 便携服务包。

原因：它是当前目标下的最小可交付闭环。对用户足够友好，对现有系统侵入足够小，对后续维护和升级也最稳妥。

## 5. 交付形态

最终交付物应为一个可拷贝、可压缩、可在 Windows 上解压即用的目录，例如：

```text
发票助手便携版/
├─ 启动发票助手.bat
├─ 停止发票助手.bat
├─ 用户指引.html
├─ README-技术维护.md
├─ app/
│  ├─ python/
│  ├─ server/
│  └─ bootstrap/
├─ data/
│  ├─ app.db
│  ├─ storage/
│  └─ exports/
├─ logs/
│  ├─ app.log
│  └─ startup.log
└─ runtime/
   ├─ app.pid
   └─ app.url
```

说明：

- `app/python/`：内置完整 `CPython 3.11 x64` 运行时目录，包含 `python.exe`、标准库和已预装的 `Lib/site-packages/`；正式交付固定采用这一种形态，不使用 embeddable Python，也不依赖目标机器上的 venv。
- `app/server/`：后端代码与前端构建产物。
- `app/bootstrap/`：启动检查、路径初始化、浏览器自动打开等辅助逻辑。
- `data/`：用户数据目录，包含数据库、上传原件、导出结果。
- `logs/`：日志目录，用于启动失败排查和运行记录。
- `runtime/`：临时运行态文件，例如 PID 和访问 URL。

## 6. 运行模型

### 6.1 单进程交付

正式交付版本不再依赖前端开发服务器。前端先执行构建，生成静态资源；后端负责提供 API 与前端静态页面。

最终用户启动后，只运行一个本地服务。

### 6.2 Windows 运行时边界

正式交付边界必须写死为：

1. 仅支持 `Windows 10/11 x64`。
2. 包内置固定版本的完整 `CPython 3.11 x64` 运行时目录，目录位置固定为 `app/python/`。
3. 包内直接携带已安装完成的 Python 依赖，不允许在用户机器首次启动时执行 `pip install` 或联网下载。
4. OCR 运行所需的 `rapidocr-onnxruntime`、`pypdfium2`、Pillow 及其模型/原生依赖必须随包交付，并可在离线环境运行。
5. 正式交付不要求用户自行安装 Python、Node、Visual Studio Build Tools 或前端工具链。
6. 交付形态固定为“完整 CPython 目录 + 已安装 site-packages + 发布态后端/前端产物”，不允许在 Windows 包内临时创建 venv，也不允许在目标机器上做解释器选择。

若最终实现无法满足以上 1-6，则不视为“便携服务包”达标。

### 6.3 访问方式

启动脚本成功后，自动打开本地浏览器访问固定地址，例如：

`http://127.0.0.1:18080`

是否使用 `18080` 不重要，关键是：

1. 端口固定。
2. 启动脚本会自动检查占用。
3. 用户无需记忆端口。

### 6.4 固定端口冲突策略

正式交付必须采用固定端口策略，不自动切换随机端口。

规则如下：

1. 若固定端口已被当前发票助手实例占用，启动脚本应识别为“已在运行”，直接打开浏览器并提示用户复用当前实例。
2. 若固定端口被其他程序占用，启动脚本必须阻断启动，不得自动改用其他端口。
3. 被其他程序占用时，启动窗口必须用中文给出明确下一步动作，例如：
   - “当前电脑上已有其他程序占用了发票助手需要的端口。”
   - “请先关闭另一个占用该端口的程序，或联系维护人员处理。”
   - “如果你只是误关了浏览器，请重新双击启动脚本即可恢复页面。”

### 6.5 发布态前端托管契约

正式交付版必须把前端构建产物作为后端静态资源托管，而不是继续运行 Vite。

发布态契约如下：

1. `/api/*` 继续由后端 API 路由处理。
2. `/health` 继续保留为健康检查入口。
3. `/assets/*` 与前端构建产物中的静态资源路径由后端直接返回。
4. `/` 返回前端构建后的 `index.html`。
5. 前端路由在浏览器刷新时，后端仍需回退到 `index.html`，避免配置页、结果页等直达时 404。
6. 后端必须通过显式配置读取发布目录中的前端静态文件位置，不能依赖开发态路径约定。

### 6.6 生产启动入口契约

正式交付必须收敛为唯一的发布态启动入口，不允许多个等价入口并存。

唯一启动入口定义为：

`app\python\python.exe app\bootstrap\start_server.py --portable-root "<便携包根目录>" --host 127.0.0.1 --port 18080`

要求：

1. `启动发票助手.bat` 只能调用上述入口，不直接拼接多条候选命令。
2. `start_server.py` 负责解析便携包根目录，并把 `data/`、`logs/`、`runtime/`、前端静态目录全部转换成绝对路径。
3. `start_server.py` 必须显式加载发布态配置，不能复用开发态下依赖 Vite 代理的默认路径。
4. 后端服务进程必须只暴露 `127.0.0.1`，不默认绑定局域网地址。

### 6.7 启动脚本职责

`启动发票助手.bat` 负责：

1. 检查 `data/`、`logs/`、`runtime/` 是否存在，不存在则初始化。
2. 检查目标数据目录与日志目录是否可写；不可写则中止启动并给出中文提示。
3. 检查目标端口是否被占用。
4. 检查 `runtime/app.pid` 是否存在；若存在，则判断对应进程是否仍在运行。
5. 若已有实例正在运行且健康，直接打开浏览器并提示“发票助手已在运行”。
6. 若 PID 文件陈旧，则先清理旧运行态文件后继续。
7. 启动本地服务。
8. 在确认 `/health` 和首页均可访问后，再写入 PID 和 URL。
9. 自动打开浏览器。
10. 以中文提示展示“已启动 / 启动失败 / 查看日志位置 / 已在运行”等信息。

此外，启动失败时，窗口文案不能只停留在技术原因，还必须告诉用户下一步做什么。最少要覆盖以下中文动作提示：

1. “请把整个发票助手文件夹移动到桌面或文档这类本地可写位置后再试。”
2. “请确认没有重复打开两个发票助手窗口。”
3. “如果还是失败，请把这个窗口截图发给维护人员。”

### 6.8 停止脚本职责

`停止发票助手.bat` 负责：

1. 读取 `runtime/app.pid`。
2. 若 PID 不存在或进程不存在，提示“当前未运行”。
3. 若进程存在，则按 Windows 进程树方式停止对应服务进程。
4. 停止成功后清理临时运行态文件。
5. 停止失败时提示日志位置。
6. 用中文提示“已停止”或“当前未运行”。

### 6.9 用户可感知的启动与停止规则

必须把以下规则写进产品文案和 `用户指引.html`：

1. 关闭浏览器页面，不等于停止发票助手。
2. 平时用完，请双击 `停止发票助手.bat` 正常关闭。
3. 不建议用户直接关闭启动黑窗来代替停止。
4. 如果只是误关浏览器，而服务仍在运行，重新双击 `启动发票助手.bat` 应直接恢复打开页面，而不是再启动一个新实例。

### 6.10 支持的安装位置

正式交付必须明确支持和不支持的安装位置。

支持：

1. 本地磁盘上的普通可写目录。
2. 含中文和空格的本地目录。
3. 非管理员用户在自己有写权限的位置运行。

不建议或不支持：

1. 网络共享盘。
2. 只读介质。
3. 受企业同步工具强占锁的目录。
4. 权限不确定、经常被系统清理的临时目录。

若检测到明显不受支持的位置，启动脚本应直接提示用户把便携包移动到本地可写目录后再运行。

## 7. 配置优先的首次使用流程

### 7.1 原则

用户第一次使用时，必须先录入公司发票信息与规则。否则系统虽然可运行，但无法按目标公司的真实标准给出可信判断。

因此，“先配置，后上传”不是建议，而是整个交付版的第一使用路径。

### 7.2 首次使用标准流程

1. 双击 `启动发票助手.bat`。
2. 浏览器自动打开首页。
3. 首页显著提示“第一次使用请先完成基础配置”。
4. 用户进入“首次配置向导”。
5. 按步骤完成：
   - 第 1 步：填写税务档案
   - 第 2 步：选择风险规则模板并按需微调
   - 第 3 步：确认命名模板
   - 第 4 步：核对当前生效配置摘要
6. 完成后返回工作台。
7. 再开始上传发票。

### 7.3 首次配置必须是向导，不是裸配置表单

正式交付版首次使用不能只把用户丢到普通配置页面自己理解字段。

要求：

1. 未完成基础配置时，首页主动作进入“首次配置向导”，而不是普通上传页。
2. 向导必须按顺序分步展示，一步只解决一类配置问题。
3. 每一步都要用业务语言解释“这一项会影响什么判断”。
4. 未完成当前步的最小必填字段前，不允许进入下一步。
5. 向导最后一步必须展示“你现在告诉系统的公司规则是什么”的摘要确认。

### 7.4 配置完成的机器判定标准

系统必须把“配置是否完成”做成明确的产品与技术门槛，而不是仅靠提示文案。

判定标准：

1. `tax_profile` 存在当前生效版本。
2. `business_rules` 存在当前生效版本。
3. `naming_rules` 存在当前生效版本。
4. 税务档案至少包含公司名称与税号等本系统当前最小必填字段。
5. 命名模板存在可用模板字符串。
6. 未满足以上条件前，工作台上传入口必须禁用，首页主按钮必须优先引导进入首次配置向导。

### 7.5 首次配置前的准备清单

`用户指引.html` 和首次启动页必须在“进入首次配置向导前”告诉用户先准备以下材料：

1. 公司全称。
2. 公司税号。
3. 常见允许自动通过的供应商名单（如适用）。
4. 需要重点复核的关键词或业务风险提示词（如适用）。
5. 希望导出后文件采用的命名格式示例。

每一项都应给业务化示例，而不是只给字段名。

### 7.6 风险规则默认模板

首次配置时必须提供可直接使用的风险规则默认模板，不能要求助理从零写规则。

至少提供两套默认模板：

1. 保守模板：更多票据进入待复核，适合第一次上线或风险口径严格的团队。
2. 常规模板：对常见低风险票据给出更积极的自动通过建议，适合日常批量处理。

模板要求：

1. 用户能先选模板，再做少量调整。
2. 每套模板都要用大白话解释区别，而不是只给参数列表。
3. 若用户不理解规则细节，使用默认模板也应能完成首次配置并开始试运行。

### 7.7 三类配置的用户意义

- 税务档案：告诉系统“什么才是本公司的发票抬头与税号”。
- 风险规则：告诉系统“哪些票据要谨慎、哪些可以直接通过、哪些要复核”。
- 命名模板：告诉系统“通过后的文件应该按什么格式自动命名”。

### 7.8 产品层引导

建议在首页顶部增加显式引导提示，只有用户完成基础配置后，这个提示才消失。提示文案应接近自然语言，而不是技术术语。

同时必须补充以下体验：

1. 未完成配置时，首页显示“还差几步完成基础配置”的状态提示。
2. 配置完成后，提供明确的“配置已完成”状态页或摘要卡片，至少展示当前公司名称、税号、当前命名样式示例和最后修改时间。
3. 配置完成页需提供“去处理发票”这一条明确下一步动作。
4. 工作台常驻展示当前正在使用的公司配置摘要，至少包含公司名称、税号尾号、当前规则模板名称和最后更新时间，避免用户忘记自己正按哪套公司规则工作。

### 7.9 导出完成后的文件可达性

导出结果不能只停留在 toast 或“导出成功”的文字提示上，必须定义成明确的产品行为。

要求：

1. 导出成功后，结果页直接显示最近一次导出的文件名与保存位置。
2. 结果页提供“打开导出文件夹”动作。
3. 若同一批次已导出过，应能在结果页看到最近导出记录，而不是需要用户去系统文件夹里自己猜。
4. `用户指引.html` 要明确告诉用户：导出后最省事的做法是点“打开导出文件夹”，而不是手动找目录。

## 8. 用户文档设计

### 8.1 面向助理的文档

文件名固定为：

`用户指引.html`

内容不应写成“系统说明书”，而应写成“第一次怎么做、平时怎么用、遇到问题怎么办”的场景式指引。

建议结构：

1. 这是什么
2. 开始前请先准备什么
3. 第一次使用前必须先做什么
4. 场景一：第一次启动并完成配置向导
5. 场景二：上传一批发票并等待处理
6. 场景三：查看系统建议、待复核、疑似重复
7. 场景四：人工复核一张票
8. 场景五：导出 ZIP 和 Excel 台账，并直接打开导出文件夹
9. 遇到问题怎么办
10. 如何安全关闭与备份数据

### 8.2 文档语气

文档必须使用技术小白可理解的中文表达。例如：

- 不说“启动本地 Web 服务”，而说“打开工具”
- 不说“配置规则模型”，而说“把你们公司的发票标准告诉系统”
- 不说“持久化目录”，而说“数据会保存在这个文件夹里”

### 8.3 文档中的强前置提醒

文档开头必须明确写出：

> 第一次使用前，请先打开首次配置向导，填写你们公司的发票信息和规则。  
> 如果这一步没做，系统无法按你们公司的标准进行可靠判断。

还必须补一段启动安全提示，明确告诉用户首次双击脚本时可能看到黑色窗口，这是正常现象；如果浏览器没有自动打开，应按什么备用方式访问页面。

文档中的“遇到问题怎么办”必须使用下一步动作句式，而不是纯原因说明。例如：

1. “请先把发票助手文件夹移动到桌面后再试。”
2. “请检查是不是已经打开过一个发票助手窗口。”
3. “请把启动窗口截图发给维护人员。”

## 9. 维护文档设计

面向维护者的文档建议命名为：

`README-技术维护.md`

应覆盖：

1. 目录结构说明
2. 启动与停止机制
3. 日志位置
4. 数据目录说明
5. 升级方式
6. 数据备份与迁移方式
7. 常见启动失败排查
8. 支持的 Windows 版本、路径要求与已知限制
9. 生产启动入口与目录解析规则
10. 升级兼容范围与失败回退方式

## 10. 数据保存与迁移策略

### 10.1 数据目录原则

所有业务数据都保存在便携包目录自己的 `data/` 下，不依赖用户目录、注册表或额外数据库服务。

### 10.2 备份方式

官方支持的最小备份方式应为：

直接复制 `data/` 目录。

前提要求：

备份前必须先正常停止服务。

### 10.3 换电脑方式

官方支持的迁移方式应为：

1. 在新电脑解压新版或同版便携包。
2. 先确认新包未运行。
3. 将旧包中的 `data/` 覆盖到新包同名目录。
4. 启动后继续使用。

### 10.4 升级原则

升级建议采用“程序目录可替换、数据目录保留”的方式，避免让助理参与任何数据库迁移或命令行操作。

### 10.5 升级兼容契约

正式交付必须明确支持的升级边界，不能把旧数据可复用当作默认推论。

建议契约：

1. 官方支持“上一正式便携版 -> 下一正式便携版”的逐版升级。
2. 跨多个正式版本跳跃升级不默认承诺，除非维护文档明确说明。
3. 升级前必须先停止服务并备份 `data/`。
4. 若程序检测到数据迁移失败，必须阻断启动，并明确提示用户恢复备份或联系维护人员。
5. 不允许在迁移失败后进入部分可用、部分损坏的灰色状态。

## 11. 构建与打包要求

### 11.1 前端交付要求

前端必须先构建为静态资源，再纳入便携包。便携版运行时不再启动 Vite 开发服务器。

### 11.2 后端交付要求

后端必须能在打包模式下：

1. 使用便携包内的数据目录。
2. 使用便携包内的存储目录。
3. 托管前端静态资源。
4. 与开发态配置解耦，不依赖 Vite 代理。
5. 只接受来自发布态启动入口注入的根目录与配置，不依赖当前工作目录猜测路径。

### 11.3 打包脚本职责

需要提供一套构建脚本，将以下内容收集进发布目录：

1. 后端代码
2. 前端 build 产物
3. 内置 Python 运行时
4. 启动/停止脚本
5. 用户指引与维护文档
6. 默认空数据目录结构

### 11.4 构建输入与输出契约

打包流程必须定义明确的输入与输出，而不是依赖手工拼目录。

建议契约：

1. 输入：`backend/`、`frontend/`、启动脚本模板、用户文档、维护文档、运行时清单。
2. 输出：`dist/invoice-assistant-windows-x64-<version>/` 发布目录。
3. 同时输出对应 zip 压缩包。
4. 输出目录内必须包含一份 manifest 或清单文件，说明交付版本、构建时间与关键依赖版本。

## 12. 验证要求

打包交付前至少需要验证：

1. 在干净的 Windows 环境中，无需预装 Python/Node 即可启动。
2. 双击启动后可以自动打开首页。
3. 用户第一次可按指引完成“先配置、后上传”的流程。
4. 上传样例发票后，主流程可完成到导出。
5. 停止脚本可以稳定关闭服务。
6. 复制 `data/` 目录后，历史数据可继续读取。
7. 在未完成基础配置前，用户无法开始批次上传。
8. 导出成功后，用户能直接定位并打开导出文件夹。
9. 首次配置必须通过向导完成，而不是裸配置页。
10. 风险规则默认模板可以让不懂规则细节的助理完成首次配置。

### 12.1 Windows 场景验收矩阵

至少补以下场景：

1. `Windows 10/11 x64`。
2. 安装目录包含中文与空格。
3. 非管理员账户运行。
4. 端口已占用。
5. 浏览器未自动打开，手动访问固定地址仍可使用。
6. 扫描票 OCR 路径可用，不只是服务可启动。
7. 使用旧 `data/` 目录覆盖升级后，历史数据仍可读取。
8. 在没有 Python、Node 的干净机器上仍可运行。
9. 从上一正式便携版升级到下一正式便携版时，旧 `data/` 目录可安全复用。

### 12.2 小白用户验收

必须做一次无口头解释的首次使用演练：

1. 提供压缩包与 `用户指引.html`。
2. 不额外口头解释。
3. 观察测试者是否能独立完成“启动 -> 配置 -> 上传 -> 导出 -> 停止 -> 备份”。
4. 测试者任何停顿、追问或误操作，都应回收为设计缺口。

## 13. 风险与处理

### 风险 1：便携包体积过大

原因：需要自带运行时与 OCR 相关依赖。  
处理：接受包体积上升，优先保证助理开箱可用。

### 风险 2：Windows 机器上的路径或权限差异

原因：助理可能把目录放在中文路径、桌面、同步盘或受限目录。  
处理：必须明确支持的安装位置约束，并在启动脚本里做可写性预检；检测失败时直接阻断启动并提示用户换到可写目录。

### 风险 3：用户先上传、后配置

原因：用户可能跳过说明直接操作。  
处理：产品层和文档层都要给出强提醒；未完成配置前必须禁用上传入口，并把首页主动作改为进入首次配置向导。

### 风险 4：用户不知道数据在哪里

原因：便携包天然容易被整包复制或误删。  
处理：在 `用户指引.html` 与 `README-技术维护.md` 中明确写出：备份前先停止服务，备份时只需要保留 `data/`。

### 风险 4A：用户导出后找不到文件

原因：非技术用户不熟悉目录结构，单靠“导出成功”提示不足以完成交付。  
处理：结果页必须展示最近导出文件与“打开导出文件夹”动作，用户文档同步说明这一入口。

### 风险 5：首次双击脚本引发安全顾虑

原因：Windows 用户可能对 `.bat`、黑色窗口、浏览器未自动打开或系统安全提示感到不安。  
处理：在 `用户指引.html` 第一屏明确说明首次启动可能看到黑色窗口，这是正常现象；若浏览器未自动打开，提供固定地址与备用操作说明；启动脚本窗口文案必须使用中文大白话，不可只输出技术日志。

### 风险 6：用户不知道当前按哪家公司的规则在工作

原因：如果长期看不到当前生效配置，用户容易误把旧规则当新规则。  
处理：工作台常驻展示当前公司配置摘要，并在配置更新后同步刷新。

## 14. 成功标准

1. 非技术用户可在 Windows 上通过双击方式启动系统。
2. 用户无须自行安装 Python、Node 或前端开发依赖。
3. 便携版正式运行时只需要一个本地服务进程。
4. 首次使用路径明确要求先配置公司发票信息与规则。
5. `用户指引.html` 能独立指导助理完成首次配置、上传、复核和导出。
6. 数据目录可直接备份和迁移。

## 15. 推荐实施顺序

1. 先把前端收成构建产物并由后端统一托管。
2. 再整理 Windows 便携目录结构与启动脚本。
3. 再补 `用户指引.html` 与 `README-技术维护.md`。
4. 最后做一次完整的 Windows 打包验收。
