# 研究结论：发票整理助手二期：真实解析与运行时加固

## 1. 研究目标

二期研究的核心问题不是“再做一套新产品”，而是把一期 MVP 中基于夹具文本、同步串行处理的能力升级为：

1. 面向真实 PDF 的解析链路。
2. 面向扫描票的本地 OCR fallback。
3. 面向 100 到 500 张批量场景的异步处理与恢复。
4. 面向失败与重试的可观察、可诊断运行时。

研究结论必须服从一期已经落地的约束：FastAPI + SQLite + 本地文件系统 + React 前端 + 本地管理员场景。

## 2. 已知约束与基线

- 一期已经建立 `Batch`、`InvoiceRecord`、`DocumentEvidence`、`ExtractedField`、`FieldCheck`、`ReviewAction`、`ExportJob` 等核心实体。
- 一期已经定义合规票总金额统计口径：仅统计 `suggested_pass` 且排除疑似重复票。
- 一期 `ProcessingService` 仍以同步方法 `process_batch()` 直接遍历票据，解析链路主要依赖夹具文本或通用正则。
- 前端已有批次工作台、结果页、详情抽屉与配置中心，二期更适合在现有轮询式进度接口上扩展，而不是推倒重来。

## 3. 关键决策

### 决策 1：继续以统一证据模型为解析边界

**结论**：保留一期的 `UnifiedDocumentEvidence -> DocumentEvidence / ExtractedField / FieldCheck` 路径，真实文本抽取和 OCR 只负责产出统一证据，不直接参与业务规则判断。

**原因**：
- 一期规则层、详情展示层和导出层已经建立在统一证据模型上。
- 真实 provider 的差异主要体现在文本块、字段候选和错误诊断，不应传播到规则引擎。
- 这能把二期工作限定在“接入 provider + 运行时加固”，而不是重写业务语义。

### 决策 2：采用“文本抽取优先，本地 OCR 兜底”的双层解析策略

**结论**：仍坚持 PRD 中的文本优先策略；当 PDF 可直接提取文本且质量足够时，不执行 OCR；只有文本不可用、质量不足或被标记为图片型 PDF 时，才进入本地 OCR。

**原因**：
- 电子票文本抽取通常比 OCR 更稳定、成本更低、速度更快。
- 扫描票和图片型 PDF 需要 OCR 兜底，但 OCR 的低置信输出不能直接替代严格业务判断。
- 这一策略与 PRD 的风险结论一致，也符合二期“尽量不引入额外复杂度”的目标。

### 决策 3：二期不引入 Redis / Celery / PostgreSQL，改为 SQLite 持久化作业 + 进程内 worker

**结论**：虽然 PRD 的推荐技术栈提过 Celery / RQ、Redis 和 PostgreSQL，但二期正式方案不采用这些组件，而是建立 SQLite 持久化作业表和进程内 worker / 协调器。

**原因**：
- 当前仓库和一期实现已经建立在 SQLite 与本地文件系统上，贸然切换到分布式组件会把“运行时加固”变成“基础设施重构”。
- 单机、本地管理员场景下，单进程单写者模型更容易保证幂等性、恢复一致性和故障定位。
- 对当前规模的批量处理，SQLite 持久化作业已经足够支撑异步化和恢复能力。

### 决策 4：进度可观察性继续使用轮询，不在二期引入 WebSocket

**结论**：继续扩展现有批次进度接口，通过更细粒度阶段和失败详情满足前端需求，不在二期新增 WebSocket / SSE 通道。

**原因**：
- 现有前端已经围绕拉取式进度构建。
- 二期的核心痛点是后台真实执行与恢复，不是实时推送本身。
- 轮询对单机场景已经足够，且对恢复后的状态重建更简单。

### 决策 5：把重试与恢复建模为独立 attempt，而不是覆盖原始失败记录

**结论**：每次单票处理、自动恢复或人工重试都生成新的 `ProcessingAttempt` 记录；`InvoiceRecord` 只指向当前有效 attempt。

**原因**：
- 只有 attempt 模型才能把“失败一次后重试成功”与“从未失败”区分开。
- 这能避免把 `failure_reason`、provider 诊断和中间状态反复覆盖，导致后续排障失真。
- 也更利于保障重试幂等性和导出统计不重复。

## 4. 不采用的方案

### 不采用 1：直接把 OCR 或文本抽取结果写成最终业务字段

**不采用原因**：会让 provider 差异直接侵入规则层与 UI 层，破坏一期已经建立的统一证据模型边界。

### 不采用 2：请求内同步跑完整批

**不采用原因**：与 PRD 的“大批量必须异步处理”相违背，也无法解决页面阻塞、服务重启恢复和失败重试问题。

### 不采用 3：失败后简单地重置整批状态重跑

**不采用原因**：整批重跑会放大重复计算和重复副作用，还会冲掉已经成功的单票结果，不利于大批量稳定性。

### 不采用 4：二期直接扩大为新一轮业务规则改造

**不采用原因**：二期主要目标是把一期语义迁移到真实执行环境。如果同时改动业务分类、重复票策略和导出口径，验证面会失控。

## 5. 冻结的技术方向

1. **真实解析边界**：以 provider adapter 输出统一证据模型。
2. **执行模型**：SQLite 持久化作业 + 进程内异步 worker / 恢复协调器。
3. **恢复策略**：服务启动时回收失联作业，把运行中状态重新推进到可恢复或可重试状态。
4. **前端交互**：继续轮询批次进度接口，补充更细粒度阶段和失败诊断。
5. **统计与导出**：维持一期合规票金额、疑似重复排除和 ReviewAction 独立留痕语义。

## 6. 执行阶段待确认项

1. 具体 PDF 文本抽取库和本地 OCR 引擎的版本选择，需要结合当前开发机和 CI 环境可安装性确认。
2. 回归样本的脱敏与存储位置需要在 execute 阶段落到仓库结构中。
3. 需要评估是否为 OCR fallback 引入图像预处理步骤，但这不阻塞二期文档冻结。
