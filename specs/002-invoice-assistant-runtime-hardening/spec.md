# 功能规格：发票整理助手二期：真实解析与运行时加固

**工作项编号**：`002-invoice-assistant-runtime-hardening`  
**创建日期**：2026-04-17  
**状态**：已完成实现与验证收口，可进入后续增强或归档决策  
**输入**：在一期 MVP 基础上，交付真实 PDF 文本提取与本地 OCR 适配、异步批处理与失败恢复能力，保持现有规则判定、金额口径、人工复核与导出口径不回退。  
**追溯来源**：`发票整理助手_评审终版_重新生成.md`、`specs/001-invoice-assistant-mvp/spec.md`

## 1. 工作项分类与目标

- **工作项类型**：`enhancement`
- **目标**：把一期基于夹具文本的同步处理链路升级为面向真实 PDF 的可恢复运行时，在不回退既有业务规则、金额统计与导出口径的前提下，支持真实文本抽取、本地 OCR 兜底、异步批处理、失败诊断和重试恢复。
- **核心约束**：
  - 保持一期结论语义不变，尤其是购方严格校验、业务分类、疑似重复降级、合规票总金额统计与人工复核留痕。
  - 继续采用单机、本地优先的部署模型，不在二期引入 Redis、分布式队列、远程对象存储或外部 OCR SaaS 作为必需依赖。
  - 解析链路仍以统一 `DocumentEvidence` 为边界，禁止把具体 provider 差异泄漏到规则层和前端展示层。
  - 二期重点是“真实解析 + 运行时加固”，不是重写业务规则，也不是把系统扩展成最终财务裁决平台。

## 2. 范围

### 2.1 覆盖范围

1. 为真实 PDF 文件接入可执行的文本抽取 provider，不再依赖夹具内嵌文本。
2. 为图片型 PDF 或文本抽取不可用的文件接入本地 OCR fallback。
3. 将真实文本抽取与 OCR 结果继续收口到统一 `DocumentEvidence`、`ExtractedField`、`FieldCheck` 数据通道。
4. 将批次处理由同步串行调用升级为可持久化的异步后台作业。
5. 增加作业级和单票级的失败诊断、错误码、可重试标记与最近失败记录。
6. 支持服务重启后的作业恢复与重试，不让批次永久停留在“处理中”。
7. 补齐回归样本，覆盖标准电子票、扫描票、低质量票、损坏或加密 PDF。
8. 保持结果页、详情页、导出和统计口径与一期一致，包括批次级与筛选级的合规票总金额展示。

### 2.2 明确不覆盖范围

1. 引入远程 OCR SaaS、远程文档解析 API 或 LLM 作为二期必选依赖。
2. 引入 Redis、Celery、RQ、PostgreSQL 或分布式任务编排。
3. 登录、权限、多角色、多租户、多主体隔离。
4. ERP / OA / 报销系统集成、自动入账或最终审批流。
5. 图片原件上传、移动端拍照采集、清单附件识别和海外票据识别。
6. 新一轮业务分类规则扩展或重新定义合规统计口径。

## 3. 用户故事与测试

### 用户故事 1：真实电子票可直接解析并保持一期判定口径（优先级：P0）

作为本机管理员，我希望上传真实电子 PDF 发票后系统直接完成文本抽取、规则判定和导出准备，以便一期已经跑通的业务结论可以进入真实样本阶段。

**优先级说明**：如果二期不能先稳定处理真实电子票，就没有资格进入 OCR fallback、异步恢复和更大批量场景。  
**独立测试**：上传真实电子 PDF 样本后，系统能完成文本抽取，输出与一期一致的字段、判定结果、详情解释和合规票金额汇总。

#### 场景

1. **Given** 管理员上传一批真实电子 PDF，**When** 批次完成处理，**Then** 结果页能展示原文件名、新文件名、金额、日期、基础合规、业务合规、最终结论和合规票总金额。
2. **Given** 某张电子票命中疑似重复且原系统判定为 `suggested_pass`，**When** 系统刷新批次统计，**Then** 该票仍被降级为 `review_required` 且不计入合规票金额。

### 用户故事 2：扫描票和图片型 PDF 可通过 OCR fallback 进入统一判定链路（优先级：P0）

作为本机管理员，我希望扫描票或图片型 PDF 在无法直接提文本时自动进入 OCR 兜底，以便系统仍能给出可解释的自动结论或待复核结论。

**优先级说明**：PRD 已明确图片型 PDF 属于支持但稳定性较低的范围，二期必须把这一路径变成真实能力。  
**独立测试**：上传扫描票样本后，系统在文本抽取失败或文本不可用时自动切换 OCR，并输出统一证据和人类可读的处理说明。

#### 场景

1. **Given** 某张扫描票无法直接提取可用文本，**When** 系统执行 OCR fallback，**Then** 该票仍进入统一字段抽取、基础合规和业务分类流程。
2. **Given** OCR 结果置信度明显不足，**When** 系统形成最终结论，**Then** 该票进入 `review_required`，而不是直接给出自动通过或自动驳回。

### 用户故事 3：大批量异步处理可观察、可恢复（优先级：P0）

作为本机管理员，我希望单批次数百张 PDF 在后台异步执行，并且页面能持续看到阶段进度和失败数量，以便我不用等待阻塞页面，也不会在服务重启后失去状态。

**优先级说明**：PRD 明确要求 100 到 500 个 PDF 异步处理；如果仍沿用同步实现，页面、服务稳定性和恢复能力都不达标。  
**独立测试**：创建较大批次后，API 返回已入队状态；后台逐张推进；服务中断后重启，作业能继续或重新入队，不出现永久卡死的批次。

#### 场景

1. **Given** 管理员上传 100 张以上 PDF，**When** 创建批次成功，**Then** API 立即返回批次与作业信息，页面通过轮询看到排队、抽取、分类、完成等阶段。
2. **Given** 服务在处理中途重启，**When** 恢复逻辑执行，**Then** 原处于运行态的作业会被重新标记为可恢复状态并继续推进，不会无限停留在“处理中”。

### 用户故事 4：失败诊断和重试对业务结果无副作用（优先级：P1）

作为本机管理员，我希望看到单票失败原因、失败阶段和是否可重试，并能对失败批次或失败票发起重试，以便我定位问题并恢复处理，而不产生重复统计、重复导出或脏数据。

**优先级说明**：没有可解释的失败和可控重试，异步批处理只会把问题从前台等待转成后台黑盒。  
**独立测试**：构造损坏 PDF、加密 PDF 或 provider 异常样本，系统记录阶段、错误码、可重试标志；重试后不会产生重复证据或重复统计。

#### 场景

1. **Given** 某张 PDF 已损坏，**When** 文本抽取或 OCR 无法继续，**Then** 该票标记为 `processing_failed`，并记录失败阶段、失败原因和是否可重试。
2. **Given** 用户对失败票发起重试，**When** 新尝试完成，**Then** 系统只保留当前有效尝试的结论与统计，不因历史失败尝试导致重复证据、重复金额或重复导出。

## 4. 边界条件

- 文本抽取返回空白、乱码或低质量结果时，必须允许切换到 OCR fallback，而不是直接判处理成功。
- OCR 输出低置信时，应优先进入 `review_required`，不能因为“字段看起来差不多”而直接判通过。
- 单张票失败不应阻塞同批次其他票继续推进，批次允许以“部分完成且含失败”的状态收口。
- 服务重启后，原本处于运行中的作业必须被恢复为可继续推进或可重试状态，不能永久停留在“extracting”“classifying”等中间态。
- 重试必须具备幂等性；同一张票多次重试不能重复累加 `suggested_pass_count`、`suggested_pass_total_amount` 或导出产物。
- 二期不得回退一期的金额展示要求：批次级和筛选级都要展示合规票总金额，且统计口径排除疑似重复票。

## 5. 功能需求

### 5.1 真实解析与证据标准化

- **FR-001**：系统必须对真实 PDF 二进制内容执行文本抽取，而不是仅处理夹具内嵌文本。
- **FR-002**：系统必须在文本抽取不可用、返回空白文本或被判定为低可用性时自动切换到本地 OCR fallback。
- **FR-003**：系统必须继续将文本抽取和 OCR 输出统一映射为 `DocumentEvidence`、`ExtractedField` 和 `FieldCheck` 所需的数据结构。
- **FR-004**：系统必须为每次解析记录 `provider_name`、`provider_version`、`source_type`、错误码、错误消息和置信度摘要。
- **FR-005**：系统必须保留一期的“低置信进入待复核”原则，不能因引入 OCR 而放宽自动通过条件。

### 5.2 异步批处理与恢复

- **FR-006**：系统必须在批次创建后持久化后台处理作业，并以异步方式推进单票处理，而不是在请求生命周期内同步跑完整批。
- **FR-007**：系统必须为批次处理记录作业状态、当前阶段、总数、完成数、失败数、心跳时间和最近失败信息。
- **FR-008**：系统必须把单票处理尝试建模为独立 attempt，以区分首次处理、恢复处理和人工重试。
- **FR-009**：系统必须保证单票失败不会阻塞同批次其他单票继续推进。
- **FR-010**：系统必须在服务启动时扫描未收口作业，把失联的运行中作业恢复到可推进状态。
- **FR-011**：系统必须让阶段推进具备幂等性，避免重复写入证据、字段或统计汇总。

### 5.3 失败诊断、重试与可观察性

- **FR-012**：系统必须对单票记录失败阶段、错误码、错误摘要、是否可重试和最近一次尝试时间。
- **FR-013**：系统必须对批次进度接口返回更细粒度的人类可读阶段文案，例如“正在抽取 PDF 文本”“正在执行 OCR”“正在校验规则”。
- **FR-014**：系统必须在结果详情或失败列表中展示解析来源、provider 信息和失败原因，而不是只给技术错误码。
- **FR-015**：系统必须支持对单票失败记录或批次失败子集发起重试。
- **FR-016**：系统必须在重试前清理或失效旧尝试的临时产物，避免重复证据和统计污染。

### 5.4 业务兼容与导出保持

- **FR-017**：系统必须保持一期的基础合规、业务分类、疑似重复检测和人工复核语义不回退。
- **FR-018**：系统必须继续在批次总览和结果筛选视角展示合规票总金额，且口径仍为“`suggested_pass` 且排除疑似重复票”。
- **FR-019**：系统必须保证 ZIP / Excel 导出的统计口径、文件归类和界面一致，不因异步化与重试而重复或漏项。
- **FR-020**：系统必须让人工复核记录继续作为独立留痕对象，不得因运行时恢复或重试覆盖历史复核动作。

### 5.5 回归样本与可配置项

- **FR-021**：系统必须建立覆盖电子票、扫描票、损坏票、加密票和低置信 OCR 票的回归样本集。
- **FR-022**：系统必须允许通过配置控制是否启用 OCR fallback、最低 OCR 置信阈值和解析 provider 选择。
- **FR-023**：系统必须把原始文件、重命名文件、解析中间产物和导出产物隔离到明确目录，避免恢复与重试时互相覆盖。
- **FR-024**：系统必须保持文档与代码可追溯，所有新增实现都能回溯到 `spec.md`、`research.md`、`data-model.md` 与 `tasks.md`。

## 6. 关键实体

- **Batch**：延续一期批次主记录，继续保存总文件数、完成数、失败数、合规票数量与总金额，并关联当前作业快照。
- **InvoiceRecord**：延续一期单票主记录，新增当前处理阶段、最近错误信息、最近尝试和重试计数等运行时字段。
- **DocumentEvidence**：保持统一解析证据边界，屏蔽真实文本抽取与 OCR 的 provider 差异。
- **ProcessingJob**：二期新增的批次级后台作业实体，记录异步处理生命周期、心跳、恢复状态和统计快照。
- **ProcessingAttempt**：二期新增的单票处理尝试实体，记录每次处理或重试的阶段、provider、错误和最终结果。
- **ProviderDiagnostic**：二期新增的解析诊断结构，可落在 `ProcessingAttempt` 或关联表中，用于对外展示 provider、错误码、错误消息和置信摘要。

## 7. 非功能要求

- **NFR-001**：二期仍必须支持单机本地运行，默认依赖控制在本机可安装的 Python 库与本地 OCR 组件范围内。
- **NFR-002**：批次处理必须异步化，前端轮询不应触发整批同步重算或长时间阻塞请求。
- **NFR-003**：系统重启后不得出现大量永久卡住的“处理中”记录；恢复策略必须可验证。
- **NFR-004**：所有失败提示必须优先面向业务用户表述，技术错误码只作为辅助信息展示。
- **NFR-005**：二期实现不得破坏一期既有 API、结果页金额统计和导出语义，除非文档显式修订并完成回归验证。

## 8. 成功标准

- **SC-001**：真实电子 PDF 样本可完成文本抽取并进入一期既有规则链路，输出稳定的字段、判定和金额汇总。
- **SC-002**：扫描票或图片型 PDF 在文本抽取不可用时能自动切换本地 OCR，并输出统一证据模型。
- **SC-003**：损坏票、加密票或 provider 异常不会阻塞整批处理，失败票会输出明确失败原因。
- **SC-004**：批次创建后立即返回异步作业信息，页面轮询可观察到阶段推进、完成数、失败数和最近失败列表。
- **SC-005**：服务重启后的恢复验证能证明未完成作业被正确回收，不存在长期卡住的运行态记录。
- **SC-006**：重试同一张票或同一失败批次后，不会重复生成证据记录、重复累计合规票金额或重复导出结果。
- **SC-007**：结果页和导出继续展示合规票总金额，且口径仍排除疑似重复票。
- **SC-008**：二期文档阶段通过 `refine`、`design`、`decompose` 和 `verify constraints` 校验，并为后续 execute 留出明确任务边界。
