**工作项编号**：`003-runtime-state-recovery`  
**创建日期**：2026-04-18  
**状态**：已完成实现与 close-out 文档收口，待后续合并 / 归档决策  
**输入**：根据 2026-04-18 对抗评审结论，优先收口运行时状态与恢复闭环：统一批次/作业/阶段语义，修正前端活跃批次识别，补启动恢复扫描与中断恢复验证，收紧最近失败列表规模，保持现有业务规则与财务治理边界不扩张。  
**追溯来源**：`发票整理助手_评审终版_重新生成.md`、`specs/002-invoice-assistant-runtime-hardening/spec.md`

## 1. 工作项分类与目标

- **工作项类型**：`hardening`
- **目标**：在二期真实解析与异步处理已经落地的前提下，补齐“运行时状态可见、恢复可追、失败列表可控”的最后闭环，避免批次页面误判空闲、恢复后的批次不可见或失败信息无限膨胀。
- **核心约束**：
  - 不扩张到权限、角色、审批、审计和财务治理新能力；三期只收口运行时闭环。
  - 保持二期已交付的真实解析、OCR fallback、批次恢复、失败重试和导出口径不回退。
  - 前后端对“活跃批次”的判断必须围绕统一阶段语义，而不是各自维护不一致的枚举。
  - 失败列表是用于操作台诊断的最近窗口，不是全量失败清单的替代接口。

## 2. 范围

### 2.1 覆盖范围

1. 统一前后端对批次活跃阶段的识别规则，覆盖 `queued`、细粒度处理中阶段和恢复中阶段。
2. 修正工作台对活跃批次的发现逻辑，确保轮询不会因为阶段码更细而停止。
3. 收紧批次进度中的最近失败窗口，只返回有限、可操作的最近失败样本。
4. 补齐进度聚合与恢复链路回归，证明服务启动恢复后批次仍能被正确识别和展示。
5. 更新三期 work item 文档、任务与执行归档，使本轮收口可追溯。

### 2.2 明确不覆盖范围

1. 登录、角色权限、审批流、审计追踪或财务裁决闭环。
2. 新增导出格式、ERP / OA 集成或外部通知。
3. 重新设计批次结果页、详情页或业务规则判定逻辑。
4. 引入新的后台队列、中间件或数据库。
5. 对全部失败记录提供新的历史查询接口；本 work item 只约束工作台诊断窗口。

## 3. 用户故事与测试

### 用户故事 1：细粒度运行阶段下工作台仍能识别活跃批次（优先级：P0）

作为操作台用户，我希望批次进入“文本抽取中”“规则校验中”“重复票检测中”等细粒度阶段时，工作台仍持续显示该批次并轮询刷新，以便我不会误以为系统已经空闲。

**优先级说明**：如果工作台在真实运行阶段下直接显示“当前没有活跃批次”，用户会失去对后台作业的可见性，影响对整条运行时链路的判断。  
**独立测试**：前端回归用例构造 `text_extraction`、`classification` 或 `recovering` 阶段批次，确认工作台展示活跃条带而不是空态。

#### 场景

1. **Given** 某批次进度阶段为 `text_extraction`，**When** 用户打开工作台，**Then** 页面展示该批次为活跃批次并显示对应阶段文案。
2. **Given** 某批次已因服务中断进入恢复队列，**When** 用户刷新工作台，**Then** 页面继续展示该批次而不是落入空态。

### 用户故事 2：最近失败窗口保持有限且按最新失败优先（优先级：P0）

作为操作台用户，我希望活跃批次只展示最近几张失败票，而不是把历史全部失败都塞进进度接口，以便页面能快速聚焦当前需要重试或诊断的样本。

**优先级说明**：最近失败窗口失控会导致接口冗长、工作台噪声过大，也会让“最近失败”这个概念失真。  
**独立测试**：后端回归用例构造多张失败票，验证进度快照只返回固定上限，并按最近失败 attempt 排序。

#### 场景

1. **Given** 某批次包含 5 张失败票，**When** 进度服务刷新快照，**Then** `recent_failures` 只返回最近 3 张失败票。
2. **Given** 一张较早失败的票和一张刚失败的票同时存在，**When** 工作台读取进度接口，**Then** 最近失败列表把刚失败的票排在更前面。

### 用户故事 3：服务启动恢复后批次状态仍与界面语义一致（优先级：P1）

作为本机管理员，我希望服务重启后的恢复逻辑不仅能重新入队作业，还能让批次继续以可观察的活跃状态出现在工作台上，以便恢复不是后台黑盒。

**优先级说明**：恢复服务本身已存在，但如果恢复后的阶段语义和前端判定不对齐，用户仍然会看到“无活跃批次”，闭环依旧断裂。  
**独立测试**：后端恢复测试结合前端活跃判定测试，证明恢复后的批次阶段码可被工作台正确消费。

#### 场景

1. **Given** 某批次在 `classification` 阶段中断，**When** 服务重启并恢复该批次，**Then** 批次被重新入队且进度阶段回到工作台认可的活跃阶段。
2. **Given** 恢复后的批次继续完成处理，**When** 进度快照刷新，**Then** 批次最终收口为完成态且不会永久停留在活跃列表。

## 4. 边界条件

- `stage_code` 为细粒度处理中阶段时，工作台不能仅因不等于字符串 `"processing"` 而判空。
- 恢复中的批次即使 `active_job_id` 暂时切换，也必须能以 `queued` 或恢复态继续显示为活跃批次。
- 最近失败窗口必须保留重试所需诊断字段，不能因为裁剪而丢失 `invoice_id`、错误码或是否可重试。
- 全部票据都失败的批次仍可收口为 `failed`；最近失败窗口裁剪不改变批次完成/失败统计。
- 三期不得把最近失败窗口误用为新的审计清单或历史归档接口。

## 5. 功能需求

### 5.1 运行时阶段语义收口

- **FR-001**：系统必须定义统一的活跃阶段集合，至少覆盖 `queued`、`processing`、`text_extraction`、`ocr_processing`、`classification`、`duplicate_check`、`finalization` 与 `recovering`。
- **FR-002**：工作台必须使用统一活跃阶段集合识别活跃批次，而不是只接受 `"queued"` 或 `"processing"` 两个阶段码。
- **FR-003**：批次恢复链路必须把恢复中的批次重置到前端可识别的活跃阶段语义，避免恢复成功但界面误判为空闲。

### 5.2 最近失败窗口控制

- **FR-004**：进度服务必须把 `recent_failures` 收敛到固定上限，默认返回最近 3 条失败记录。
- **FR-005**：`recent_failures` 必须按最近一次失败 attempt 的时间优先级排序，最新失败排在最前。
- **FR-006**：`recent_failures` 的裁剪不得丢失工作台和重试入口依赖的字段，包括 `invoice_id`、`original_filename`、`failure_reason`、`failure_stage`、`error_code`、`retryable`、`parse_source` 与 `provider_diagnostic`。

### 5.3 验证与追溯

- **FR-007**：系统必须补齐针对细粒度阶段活跃识别的前端回归测试。
- **FR-008**：系统必须补齐针对最近失败窗口上限与排序的后端回归测试。
- **FR-009**：三期 work item 文档必须明确实现边界、任务批次和验证命令，并与本次实现保持可追溯。

## 6. 关键实体

- **BatchProgressSnapshot**：批次进度聚合结果，负责向前端输出当前阶段、统计汇总和最近失败窗口。
- **ProcessingJob**：后台作业实体，提供当前阶段、恢复态和运行中语义来源。
- **ProcessingAttempt**：单票处理尝试实体，提供最近失败排序所需的阶段和时间信息。
- **BatchWorkbench**：工作台页面，消费 `BatchProgressSnapshot` 并决定是否将批次视为活跃。

## 7. 非功能要求

- **NFR-001**：三期修复必须保持现有 SQLite + 本地文件系统 + 进程内 worker 架构，不新增运行时基础设施。
- **NFR-002**：工作台活跃批次判断必须是纯前端轻量逻辑，不能引入额外 API 往返。
- **NFR-003**：最近失败窗口裁剪后，进度接口响应体规模应与活跃诊断用途一致，不随批次失败总量线性膨胀。
- **NFR-004**：三期验证优先通过现有后端 pytest 与前端 vitest 回归完成，不以人工描述代替证据。

## 8. 成功标准

- **SC-001**：当前端收到 `text_extraction`、`classification`、`duplicate_check`、`finalization` 或 `recovering` 阶段批次时，工作台不再显示“当前没有活跃批次”。
- **SC-002**：`recent_failures` 在失败票数量超过 3 时仍只返回 3 条，并按最新失败优先排序。
- **SC-003**：现有恢复测试继续通过，并能证明服务启动恢复不会破坏活跃批次可见性。
- **SC-004**：`spec.md`、`plan.md`、`tasks.md` 与执行日志可回溯本次收口范围，不保留脚手架占位文本。
