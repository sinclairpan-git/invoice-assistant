# 功能规格：导出审计回显收口

**功能编号**：`009-export-audit-surface-refresh`
**创建日期**：2026-04-19
**状态**：已完成实现与归档收口（2026-04-19）
**输入**：结果页导出成功后应刷新并展示持久化导出状态，收口 export_manifest_path/export_jobs 的用户可见审计回显。 参考：`docs/invoice-assistant-gap-backlog.zh-CN.md`、`specs/004-controlled-review-export/spec.md`、`docs/superpowers/specs/2026-04-17-invoice-assistant-design.md`

**范围**：

- 覆盖批次结果页导出成功后的前端刷新行为。
- 覆盖批次结果页对 `export_manifest_path` 与 `export_jobs` 的用户可见回显。
- 覆盖受影响前端测试断言、顶层 backlog 与本 work item 归档。
- 不覆盖后端导出契约、审计落库逻辑、导出文件内容和权限门槛。
- 不覆盖新的导出类型、下载流程改造或独立导出历史页面。

## 用户场景与测试（必填）

### 用户故事 1 - 导出成功后结果页要保留可见审计痕迹（优先级：P1）

作为导出责任人，我希望在结果页执行导出后，页面能立即刷新并展示最新的导出状态和台账路径，这样我离开 toast 之后仍能确认这次导出到底留下了什么产物。

**优先级说明**：当前后端已经保存了导出作业与 manifest 路径，但前端成功导出后只显示一次性 toast，导致用户可见审计链路在结果页断开。这不是权限或数据正确性的 blocker，但它会削弱“导出后审计完整”的可感知闭环。

**独立测试**：前端运行时 UI 测试锁定“导出 Excel 台账后页面刷新并显示最近导出与持久化路径”。

**验收场景**：

1. **Given** 某批次当前还没有导出回显，**When** 用户在结果页点击“导出 Excel 台账”且导出成功，**Then** 页面会刷新批次详情，并展示“最近导出”与最新的 manifest 路径。
2. **Given** 后端已经返回 `export_jobs` 与 `export_manifest_path`，**When** 结果页重新渲染，**Then** 用户能在不离开结果页的情况下看到导出类型、状态和持久化导出路径。

---

### 边界情况

- 若批次尚未导出，结果页不应凭空显示空的“最近导出”区块。
- 若 `export_manifest_path` 与某个导出作业的 `output_path` 相同，页面应避免重复展示同一路径。
- 本轮只消费现有后端字段，不新增轮询或独立审计查询接口。

## 需求（必填）

### 功能需求

- **FR-001**：批次结果页在任一导出成功后必须主动刷新批次详情，避免页面停留在导出前状态。
- **FR-002**：批次结果页必须把后端已返回的 `export_manifest_path` 与 `export_jobs` 展示为用户可见的导出回显。
- **FR-003**：导出回显至少应包含“最近导出”区块、导出类型、导出状态以及持久化导出路径。
- **FR-004**：当 `export_manifest_path` 与导出作业路径相同，页面必须避免重复输出同一路径。
- **FR-005**：本轮不得改动后端导出 API、导出服务或审计落库逻辑。

### 关键实体（如涉及数据则必填）

- **Batch.export_manifest_path**：当前批次最近一次 Excel 台账导出的持久化路径。
- **Batch.export_jobs**：当前批次已有的导出作业摘要，用于结果页回显导出类型与状态。

## 成功标准（必填）

### 可度量结果

- **SC-001**：导出成功后，结果页无需手动刷新也能展示“最近导出”区块。
- **SC-002**：前端测试稳定断言最近导出回显与持久化路径，不再只依赖 toast 反馈。
- **SC-003**：本轮实现不引入后端契约变更，现有导出 API 与测试语义保持兼容。
