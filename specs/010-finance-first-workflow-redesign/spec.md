# 功能规格：财务优先工作流重设计
**功能编号**：`010-finance-first-workflow-redesign`
**创建日期**：2026-04-24
**状态**：规格已冻结（2026-04-24，已通过两名对抗 agent 合议）
**输入**：基于整体验收反馈，重构发票整理助手的信息架构、配置中心、批次结果、人工确认与导出主流程，确保后续配置与首次配置一致、界面使用财务语义，并以“另存为”替代自动落盘。参考：`docs/invoice-assistant-gap-backlog.zh-CN.md`、`docs/superpowers/specs/2026-04-24-invoice-assistant-ux-redesign.md`

**范围**：
- 覆盖首次配置成功后的再次调整体验，要求与首次配置保持相同字段心智，而不是暴露 JSON 编辑器。
- 覆盖批次创建、批次结果、待人工确认、发票详情与导出主流程的财务语义重设计。
- 覆盖“本批次重复”“人工确认即时生效”“处理中的等待态”“只在用户显式另存为时导出”的产品边界。
- 覆盖后端配置模型、状态模型、人工复核返回契约与预览接口边界的收敛要求。
- 覆盖本 work item 的正式规格、实施计划、任务拆解与合议归档。
- 不覆盖 OCR/版式识别算法重写；本轮只要求让正常识别结果能被正确消费与呈现。
- 不覆盖历史批次数据回写；历史数据仅做只读兼容。
- 不覆盖权限系统、多角色审批链和远程部署能力。

## 用户场景与测试（必填）
### 用户故事 1 - 后续配置必须像首次配置一样填写字段（优先级：P0）
作为财务配置维护人，我希望首次配置完成后再次调整公司税务档案、审核策略和命名规则时，仍然是同样的字段表单，而不是 JSON 文本编辑，这样我不需要理解系统内部键名也能安全修改。

**优先级说明**：当前“首次配置友好、后续配置技术化”是最明显的反人类体验，直接阻断非技术财务用户独立维护系统。

**独立测试**：UI 回归测试验证“配置中心”展示字段表单，不再展示 JSON 编辑器；保存前出现变更确认页。

**验收场景**：
1. **Given** 用户已完成首次初始化，**When** 打开“配置中心 > 公司税务档案”，**Then** 看到与首次配置一致的字段项，而不是原始 JSON。
2. **Given** 用户修改了税号或开户地址，**When** 点击保存，**Then** 进入“发布新版本确认”页，看到变更摘要、变更原因和变更前后对比。

---

### 用户故事 2 - 批次结果页必须围绕财务动作组织（优先级：P0）
作为批次处理人，我希望批次结果页先告诉我哪些建议通过、哪些待人工确认、哪些是本批次重复、哪些需要补充或重试，并提供“另存为”动作，而不是把内部状态和诊断信息直接摆在主表里。

**优先级说明**：这是最高频页面。若结果页仍以系统状态为中心，用户就必须先理解系统内部概念才能完成整理动作。

**独立测试**：浏览器回归测试验证结果页存在“建议通过 / 待人工确认 / 本批次重复 / 需补充/重试”分组，以及“另存为当前筛选结果 / 另存为勾选结果”动作；页面不再自动落盘导出。

**验收场景**：
1. **Given** 一个批次处理完成，**When** 用户进入结果页，**Then** 顶部先显示按财务动作分组的摘要，而不是原始风控标签。
2. **Given** 用户尚未决定是否导出，**When** 浏览批次结果，**Then** 系统不会自动把重命名文件落到项目目录。

---

### 用户故事 3 - 人工确认必须成为独立工作流（优先级：P1）
作为人工复核人，我希望系统把待确认发票集中到独立队列里，并在我确认通过或驳回后立即刷新列表与详情状态，这样我可以连续处理，而不是反复手工刷新和猜测是否成功。

**优先级说明**：人工确认是第二高频路径。当前“抽屉底部补丁式提交”让复核不可批量、不可追踪，也容易造成状态不同步。

**独立测试**：UI 回归测试验证待人工确认队列、单票确认动作、确认后列表和详情即时同步。

**验收场景**：
1. **Given** 某张发票进入人工确认队列，**When** 复核人点击“确认通过”，**Then** 当前详情、结果列表和批次摘要都立即显示已确认结果。
2. **Given** 用户只想查看待人工确认发票，**When** 打开独立确认页，**Then** 页面默认只显示需要人工处理的票据。

---

### 用户故事 4 - 重复判定只看当前批次（优先级：P0）
作为批次处理人，我希望“重复发票”只基于当前批次判定，而不是跨历史批次拦截，这样历史归档不会影响我当前的临时试传或重复上传检查。

**优先级说明**：用户已经明确否定“历史批次重复拦截”的产品语义，必须在正式规格中冻结边界。

**独立测试**：后端回归测试验证第二个批次上传历史已出现的发票时，不会因为历史记录而标记重复；同一批次内重复仍能标记。

**验收场景**：
1. **Given** 历史批次存在同一发票，**When** 用户在新批次再次上传，**Then** 系统不会把它标为重复。
2. **Given** 同一张发票在当前批次上传两次，**When** 处理完成，**Then** 系统会在结果页把它归到“本批次重复”。

---

### 边界情况

- 批次刚创建、识别尚未完成时，结果页必须显示“处理中 / 等待识别”，不得秒报“税号缺失”或“未识别”。
- PDF 预览必须提供真实可用的预览资源；若预览资源生成失败，必须给出可理解的回退信息和原始文件入口。
- 高级诊断信息可以保留，但默认折叠到二级区域，不得占据详情首屏。
- 历史配置版本只读展示，不允许直接编辑历史 JSON 并回写当前配置。
- 兼容期内旧导出接口可以保留，但不得继续作为主流程入口。

## 需求（必填）
### 功能需求
- **FR-001**：首次配置与后续配置必须复用同一套字段表单组件和字段命名，不得再向财务用户暴露原始 JSON 编辑入口。
- **FR-002**：配置中心必须至少拆分为“公司税务档案”“审核策略”“文件命名”“历史变更”四个区域。
- **FR-003**：配置保存必须进入发布确认流程，展示变更摘要、变更原因、变更前后对比和影响范围提示。
- **FR-004**：批次结果页必须使用财务动作分组：建议通过、待人工确认、本批次重复、需补充/重试、全部。
- **FR-005**：批次结果页默认不得展示内部技术字段，如 `risk_flags`、`provider_error`、`display_status` 等作为主列。
- **FR-006**：系统不得在用户未明确触发导出前自动把重命名结果落盘；主流程只提供“另存为当前筛选结果”“另存为勾选结果”和台账导出。
- **FR-007**：重复判定必须限定在当前批次，不得以历史批次数据作为重复拦截依据。
- **FR-008**：人工确认必须提供独立入口，并在确认通过、驳回或暂不处理后立即更新当前票据、结果列表和批次摘要。
- **FR-009**：结果页与详情页必须支持稳定的 PDF 预览；预览接口与前端渲染边界应清晰，不再由后端直接返回 HTML 页面作为主预览形式。
- **FR-010**：结果页在识别未完成前必须维持处理中状态，不得将“尚未识别完成”误判成最终失败原因。
- **FR-011**：后端配置接口必须向前端提供 typed schema 或等价的稳定对象模型，不得继续以无结构 JSON 作为主要 UI 契约。
- **FR-012**：状态模型必须收敛为稳定的处理状态、复核状态、归档状态；用户可见文案由前端或展示层派生，不得再以中文状态文案作为主逻辑判断键。
- **FR-013**：`ConfigBundle` 必须成为当前生效配置的唯一写入事实源；发布时使用统一的 bundle 版本号递增，旧的 `tax_profile`、`business_rules`、`naming_rules` 版本流只作为兼容读取或内部投影，不再作为独立主写入口。
- **FR-014**：稳定状态模型必须冻结合法枚举、状态转移规则以及与“另存为”行为的关系；其中 `archive_status` 表示业务上是否可另存为及是否已完成显式另存为，不得表示自动落盘结果。
- **FR-015**：人工确认必须采用持久化 `ReviewQueueItem` 作为队列的唯一事实源；每张发票在同一批次最多存在一个打开中的工作项，工作项需冻结入队条件、出队条件、排序规则、幂等规则、并发冲突处理，以及复核提交后返回的权威摘要对象。
- **FR-016**：结果页的 canonical business bucket 固定为“建议通过 / 待人工确认 / 本批次重复 / 需补充/重试”；技术上的 `processing_status=failed`、缺附件、票面异常等都作为 `需补充/重试` 的次级原因，不再作为并列主分组。

### 关键实体（如涉及数据则必填）

- **ConfigBundle**：统一配置方案对象，是当前生效配置的唯一写入事实源，至少包含 `profile`、`review_policy`、`naming_policy`、`bundle_version_no`、`changed_by`、`changed_at`、`change_summary`、`change_reason`。旧三类配置版本流在兼容期内只作为内部拆分存储或只读映射。
- **BatchActionSummary**：面向结果页的批次摘要对象，固定包含“建议通过”“待人工确认”“本批次重复”“需补充/重试”的 canonical business bucket 统计；技术失败、材料缺失等只作为 `需补充/重试` 的次级原因码或说明文本出现。
- **ReviewQueueItem**：人工确认持久化工作项，是人工确认队列的唯一事实源。每张发票在同一批次最多存在一个 `open` 工作项；当发票满足“处理完成且系统要求人工确认”时入队，在“确认通过 / 确认驳回 / 重新处理后不再需要人工确认”时出队。重复提交同一复核结论应幂等，冲突更新由版本号或时间戳校验拦截；复核接口提交成功后必须返回更新后的单票数据和权威 `BatchActionSummary`。
- **StableInvoiceStatus**：由 `processing_status`、`review_status`、`archive_status` 组成的稳定状态集合。最小枚举为：`processing_status ∈ {queued, processing, completed, failed}`；`review_status ∈ {not_required, pending, approved, rejected}`；`archive_status ∈ {not_ready, ready_to_save, saved}`。其中 `ready_to_save` 表示业务上已具备另存为条件但尚未完成显式另存为，`saved` 表示用户已通过“另存为”完成一次明确导出动作。

## 成功标准（必填）

### 可度量结果
- **SC-001**：后续配置页面主路径不再出现原始 JSON 编辑器。
- **SC-002**：首次配置与配置中心复用同一套字段表单，相关 UI 回归测试通过。
- **SC-003**：批次结果页默认使用财务动作语言组织信息，浏览器回归测试能断言“建议通过 / 待人工确认 / 本批次重复 / 需补充/重试”。
- **SC-004**：系统不再在主流程中自动落盘重命名结果；另存为下载链路在浏览器回归和接口测试中通过。
- **SC-005**：人工确认提交后，无需手动刷新即可看到列表、详情和批次摘要同步更新。
- **SC-006**：新批次上传历史已出现的发票时，不会被历史记录判为重复；当前批次内部重复判定保持有效。
- **SC-007**：结果页在解析未完成期间显示处理中状态，不再秒报“税号缺失”。
- **SC-008**：人工确认工作台上线时，用户已能稳定预览 PDF 或看到明确的原始文件入口，不需要盲审。
