# 实施计划：清单附件识别闭环

**编号**：`005-attachment-list-recognition` | **日期**：2026-04-18 | **规格**：specs/005-attachment-list-recognition/spec.md

## 概述

本期在现有本地批量处理架构上补齐“详见清单”发票的附件识别闭环。核心目标不是新增一个独立附件模块，而是让批次上传、解析、风险分类、结果展示、导出解释五条现有主路径都能理解清单附件，从而减少本可自动判断的 `review_required`。

## 技术背景

**语言/版本**：Python 3.12、TypeScript、React
**主要依赖**：FastAPI、SQLAlchemy、Pydantic、Ant Design、Vitest、Pytest
**存储**：本地 SQLite + 文件系统存储目录
**测试**：`pytest` 后端集成/单元测试，`vitest` 前端组件与 API 状态测试
**目标平台**：本地单机桌面/浏览器工作台
**约束**：

- 继续复用现有 `ProcessingService`、统一证据模型和 `risk_classifier`，不引入第二套分类引擎。
- 不接入外部 OCR、ERP/OA、报销系统，也不扩展多用户权限模型。
- 005 必须兼容 004 的受控导出与审计解释口径。

## 宪章检查

| 宪章门禁 | 计划响应 |
|----------|----------|
| 本地优先、单机闭环 | 所有附件解析、匹配、落库和展示都在本地完成，不增加外部服务依赖。 |
| 可解释自动化 | 对每个附件匹配结论记录依据、失败原因和对系统建议的影响，确保复核人可追溯。 |
| 小步迭代、主路径稳定 | 先补文档和红灯测试，再在现有批次上传与分类链路上增量扩展，保留无附件发票的现有行为。 |
| 导出治理不回退 | 所有新增证据字段都沿用 004 已固定的导出/审计边界，不新增绕过角色控制的旁路。 |

## 阶段计划

### Phase 0：研究与决策冻结

**目标**：把 005 的范围、非目标、关键数据实体和验证口径写实，替换 `workitem init` 生成的通用模板。  
**产物**：spec.md / plan.md / tasks.md / task-execution-log.md  
**验证方式**：文档自检 + `python -m ai_sdlc run --dry-run`  
**回退方式**：仅文档变更，可通过单次 git 回滚撤销。

### Phase 1：批次上传与数据模型扩展

**目标**：让批次上传入口和持久化层能够存储附件文件、附件识别状态和主发票关联关系。  
**产物**：后端模型迁移、批次创建入口、序列化结构、上传接口测试  
**验证方式**：后端红灯/绿灯测试 + 批次 API 定向测试  
**回退方式**：删除新增字段/表并回退上传接口变更。

### Phase 2：附件解析、匹配与分类复用

**目标**：把清单附件纳入统一证据模型和风险分类输入，减少“详见清单”默认待复核。  
**产物**：附件解析器、匹配规则、分类器扩展、规则测试夹具  
**验证方式**：`pytest` 规则与处理链路测试  
**回退方式**：保留附件存储但禁用分类参与逻辑，恢复到仅展示附件状态。

### Phase 3：结果展示与导出对齐

**目标**：在结果页、发票详情和导出摘要中展示附件证据链，并维持 004 的导出治理边界。  
**产物**：前端类型/API 适配、上传面板提示、详情抽屉展示、导出摘要字段、前端测试  
**验证方式**：`vitest` 组件/API 测试 + 端到端手工走查  
**回退方式**：隐藏新增 UI 字段，保留后端能力以便后续继续迭代。

## 工作流计划

### 工作流 A：上传与识别入口

**范围**：`batches.create`、文件存储、批次序列化、前端上传表单  
**影响范围**：批次创建 API、上传组件、批次详情结构  
**验证方式**：创建批次接口测试、上传表单测试、手工上传混合样本  
**回退方式**：回退到仅接受发票文件的上传参数协议

### 工作流 B：附件证据进入分类主路径

**范围**：`ProcessingService`、附件解析、匹配规则、`risk_classifier`  
**影响范围**：统一证据模型、业务分类结果、待复核原因  
**验证方式**：规则测试、处理链路测试、样本批次回归  
**回退方式**：关闭附件参与分类，只保留附件状态记录

### 工作流 C：结果解释与导出

**范围**：API serializer、导出摘要、结果页详情抽屉  
**影响范围**：复核体验、受控导出说明、审计可追溯字段  
**验证方式**：序列化测试、前端快照/交互测试、导出字段对账  
**回退方式**：隐藏新增字段并保留底层数据

## 关键路径验证策略

| 关键路径 | 主验证方式 | 次验证方式 |
|----------|------------|------------|
| 混合上传批次能正确识别附件候选 | 批次 API 集成测试 | 浏览器手工上传 |
| “详见清单”发票在有可信附件时不再默认待复核 | 规则与处理链路测试 | 样本批次回归 |
| 附件异常不会误导成自动通过 | 负向测试夹具 | 手工比对详情与导出 |
| 导出与结果页都能解释附件证据 | serializer/UI 测试 | 手工导出检查 |

## 开放问题

| 问题 | 状态 | 阻塞阶段 |
|------|------|----------|
| 附件候选的判别是否只支持 PDF，还是要允许现有 OCR 路径处理图片页 PDF | 已收敛为“先支持 PDF 文件，图片页由 PDF 内页 OCR 兜底” | Phase 1 |
| 一张发票对应多份附件时是否需要人工排序 | 本期不做人工排序，只聚合可信附件并记录顺序 | Phase 2 |
| 导出摘要里新增字段的粒度是完整清单摘要还是精简原因码 | 先落精简摘要与原因码，避免导出表头膨胀 | Phase 3 |

## 实施顺序建议

1. 先冻结 005 文档与任务分批，避免继续沿用脚手架占位内容。
2. 先写后端红灯测试，覆盖“有清单可自动判断”和“无清单仍待复核”两条主路径。
3. 增量扩展批次上传、数据模型、附件解析与匹配。
4. 在分类器里接入附件证据，随后补齐结果页与导出说明。
5. 最后做无附件回归、混合上传走查和 AI-SDLC dry-run 验证。
