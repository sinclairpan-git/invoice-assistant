# 研究记录：发票整理助手一期 MVP

## 1. 研究目标

本文件用于冻结第一期的关键技术和产品实现决策，确保后续 `plan.md`、`tasks.md` 和实现阶段以同一套基线推进。

## 2. 约束输入

- 用户明确要求遵循当前仓库的 AI-SDLC 流水线，不把对话内宿主 workflow 误当作 execute 授权。
- PRD 与辅助设计文档已经收敛出第一期唯一生效基线：单机、本地 Web、单管理员、文本提取优先、本地 OCR 兜底。
- 第一阶段目标是设计冻结与分解，不是立即编写产品代码。

## 3. 关键方案比较

### 3.1 本地 Web 应用 vs 桌面打包应用

| 方案 | 结论 | 原因 |
|------|------|------|
| 本地 Web 应用（浏览器访问本地服务） | 采用 | 与 PRD 收敛结论一致，开发与调试成本低，便于前后端分离和后续扩展 |
| Electron / Tauri 桌面应用 | 不采用 | 第一阶段没有强需求，打包和运行时复杂度更高 |

### 3.2 React + Vite + Ant Design vs Next.js + shadcn/ui

| 方案 | 结论 | 原因 |
|------|------|------|
| React + Vite + TypeScript + Ant Design | 采用 | 页面以管理台和数据表格为主，Ant Design 更贴合批次工作台、结果页、表单配置中心等场景 |
| Next.js + shadcn/ui | 不采用 | 与第一期单机本地、低部署复杂度目标不匹配，且已被设计基线显式排除 |

### 3.3 FastAPI + SQLAlchemy + SQLite vs PostgreSQL + Celery + Redis

| 方案 | 结论 | 原因 |
|------|------|------|
| FastAPI + SQLAlchemy + SQLite | 采用 | 足以支撑单机、本地、单写者模型；依赖少，易启动，便于保留审计留痕 |
| PostgreSQL + Celery + Redis | 不采用 | 超出第一期复杂度预算，且不符合当前单机运行约束 |

### 3.4 文本提取优先 vs OCR 先行

| 方案 | 结论 | 原因 |
|------|------|------|
| 文本提取优先，OCR 兜底 | 采用 | 对电子 PDF 成本更低、稳定性更高，且能减少 OCR 噪音 |
| OCR 全量先行 | 不采用 | 会增加识别误差和本地资源消耗，不适合第一期 |

## 4. 冻结决策

### 4.1 统一证据模型先于规则判断

所有解析能力先输出 `DocumentEvidence`，再进入字段抽取、购方校验、风险分类、重复票检测、命名和导出。这样后续更换 OCR 或文本提取库时，不需要重写整个处理链路。

### 4.2 单写者模型

后台任务允许解析步骤内部并发，但数据库写入和批次状态推进采用单写者模型。原因是：

1. SQLite 在单机场景下最适合串行写入。
2. 批次统计、审计留痕和规则快照更容易保持一致。
3. 第一阶段更需要确定性与可恢复性，而不是追求极限吞吐。

### 4.3 业务风险分类只做有限规则集

第一期只实现有限白名单、黑名单与兜底待复核规则，不建立通用 DSL 或规则平台。这样可以在较低复杂度下覆盖核心价值场景，并保留未来扩展空间。

### 4.4 疑似重复票在第一期即前置处理

重复票检测不再推迟到增强阶段，而是进入第一期主链路，并直接影响展示状态和金额统计。理由是用户已经明确要求“合规的票，要展示个总金额”，如果不先排除重复票，金额统计会失真。

### 4.5 人工复核不回写系统判定

人工动作只更新 `review_status` 与 `ReviewAction`，不覆盖 `system_decision`。这样可以同时保留系统判断和人工判断，避免历史审计失真。

## 5. 实现含义

1. 后端需要建立清晰的领域边界：文件存储、解析适配、规则判断、配置版本、导出与审计留痕。
2. 前端首页必须是批次工作台，而不是单纯上传页。
3. 结果页需要同时支持统计、筛选、详情解释和人工复核入口。
4. 配置中心虽然只有单管理员，但必须保留操作者名、版本号、批次规则快照。
5. 统计口径必须统一使用“`system_decision = suggested_pass` 且 `duplicate_flag = false`”。

## 6. 实施前仍需确认但不阻塞 design 的事项

1. 本地 OCR 供应商的具体落地包：优先通过适配器接口封装，在 execute 阶段再根据环境可用性选定。
2. 样例 PDF 数据集：需要在 execute 前准备最少一组标准电子票、一组扫描件和一组疑似重复样本。
3. Excel 导出样式深度：第一期先保证字段完整和统计正确，样式美化不作为阻塞项。

## 7. 结论

第一期方案已经具备进入 `design -> decompose -> verify` 的条件。接下来的重点不是继续争论技术路线，而是按 canonical 文档中的任务分批实现，并在获得明确 execute 授权后开始产品代码开发。
