# 功能规格：发票整理助手一期 MVP

**工作项编号**：`001-invoice-assistant-mvp`  
**创建日期**：2026-04-17  
**状态**：已完成 MVP 实现与 close-out 收口，待后续合并 / 归档决策  
**输入**：基于 PRD 交付单机本地版发票整理助手第一期，包含批次处理、购方校验、业务风险分类、重复票检测、重命名、导出、配置中心与审计留痕。  
**追溯来源**：`发票整理助手_评审终版_重新生成.md`、`docs/superpowers/specs/2026-04-17-invoice-assistant-design.md`

## 1. 工作项分类与目标

- **工作项类型**：`new_requirement`
- **目标**：交付一个单机本地运行的发票整理助手，支撑企业内部报销初审场景中的批量 PDF 发票整理、规则校验、风险分类、人工复核和导出。
- **核心约束**：
  - 单管理员、本机浏览器访问本地服务，无登录流程。
  - 文本提取优先，本地 OCR 兜底，并通过统一证据模型隔离解析供应商差异。
  - 一期 MVP 已完成 canonical 文档、执行计划、实现与 close-out 收口；后续增强必须通过新的 work item 承接。

## 2. 范围

### 2.1 覆盖范围

1. 批量上传 PDF 发票并创建批次。
2. 展示批次处理进度、处理数量、失败数量和系统建议通过统计。
3. 文本提取优先，本地 OCR 兜底。
4. 统一 `DocumentEvidence` 证据模型与字段置信度记录。
5. 购方字段严格校验，支持历史合法别名。
6. 业务风险分类，输出系统建议通过、系统建议驳回、待复核。
7. 本机历史库内的疑似重复票检测。
8. 默认重命名规则 `YYYYMMDD_金额_发票号码.pdf`。
9. 批次结果页筛选、详情抽屉、问题解释与最小人工复核。
10. 系统建议通过金额汇总，且统计口径排除疑似重复票。
11. ZIP / Excel 导出。
12. 配置中心、版本化、批次规则快照、人工动作留痕。

### 2.2 明确不覆盖范围

1. 登录、权限、多角色体系。
2. 真伪联网查验、作废校验、红冲联网校验。
3. 图片直接上传和移动端拍照采集。
4. 多公司主体、多机部署、多人实时协作。
5. ERP / OA / 报销系统集成。
6. 通用规则平台和复杂审批流。

## 3. 用户故事与测试

### 用户故事 1：批次工作台与批量处理（优先级：P0）

作为本机管理员，我希望一次上传一批 PDF 发票并实时查看处理进度，以便我能快速判断这一批发票是否已进入可复核状态。

**优先级说明**：没有批次工作台就无法承接主流程，也无法驱动后续识别、校验、导出与复核。  
**独立测试**：上传多张 PDF 后，系统能创建批次、落盘原始文件、展示阶段进度与统计。

#### 场景

1. **Given** 管理员上传一批合法 PDF，**When** 系统建立批次，**Then** 页面展示批次编号、总文件数、处理中数量和完成进度。
2. **Given** 某张发票文本提取失败且 OCR 也失败，**When** 批次继续执行，**Then** 该记录标记为处理失败且不阻塞同批次其他发票。

### 用户故事 2：结果判定、解释与金额汇总（优先级：P0）

作为本机管理员，我希望看到系统对每张票的建议结论、依据和汇总金额，以便我能快速筛出可直接采纳和必须复核的发票。

**优先级说明**：结果页是业务价值的主要出口，也是“总金额”和“问题解释”的唯一承载面。  
**独立测试**：处理完成后，结果页能按显示状态筛选，并在批次级和筛选级展示系统建议通过金额。

#### 场景

1. **Given** 某批次中存在建议通过、建议驳回、待复核和疑似重复票，**When** 管理员进入结果页，**Then** 系统按显示状态分组，并展示对应数量统计。
2. **Given** 某张票被识别为疑似重复且原系统判定为建议通过，**When** 页面计算系统建议通过数量与金额，**Then** 该票不计入系统建议通过统计。

### 用户故事 3：配置中心与规则版本留痕（优先级：P0）

作为本机管理员，我希望维护公司税务档案、风险分类规则和命名模板，并让每个批次绑定当时的规则版本，以便历史结果可追溯。

**优先级说明**：没有配置版本和规则快照，系统结论无法解释，也无法支撑后续审计。  
**独立测试**：修改任一配置项后系统生成新版本；新建批次时记录所用版本；历史批次不受后续配置修改影响。

#### 场景

1. **Given** 管理员更新了公司税务档案，**When** 保存成功，**Then** 系统生成新的配置版本并记录变更人、时间和变更说明。
2. **Given** 某批次在旧规则版本下完成处理，**When** 后续又修改命名模板，**Then** 旧批次仍保留原快照和原统计口径。

### 用户故事 4：人工复核与导出（优先级：P1）

作为本机管理员，我希望对待复核或疑似重复票做最小人工复核并导出结果，以便我能把建议通过票和问题票分开处理。

**优先级说明**：人工复核和导出决定了第一期能否真正进入业务使用，但依赖前述处理链路先稳定落地。  
**独立测试**：在详情抽屉完成人工复核动作后，系统记录复核日志；导出 ZIP / Excel 时按指定口径输出。

#### 场景

1. **Given** 某张票处于待复核，**When** 管理员在详情抽屉中执行人工确认通过并填写备注，**Then** 系统记录复核动作但不覆盖原始系统判定字段。
2. **Given** 管理员导出系统建议通过 ZIP 和 Excel 台账，**When** 导出完成，**Then** ZIP 与 Excel 的统计口径都排除疑似重复票。

## 4. 边界条件

- 发票某个购方字段未出现时，该字段不参与严格匹配，不应直接触发驳回。
- 发票字段出现但置信度较低时，应优先进入待复核，而不是直接判定通过。
- 详见清单但缺少清单、明细仅为“商品”或“服务费”、混合票、明细抽取失败时，应进入待复核。
- 金额、日期或发票号码任一缺失时，不自动重命名，但原始文件仍需保留并可导出。
- 处理失败、疑似重复、待复核都需要在界面上可解释，不能仅以错误码呈现。

## 5. 功能需求

### 5.1 批次与文件处理

- **FR-001**：系统必须支持批量上传 PDF 文件并创建批次。
- **FR-002**：系统必须在批次层记录总文件数、完成数、处理中数、失败数。
- **FR-003**：系统必须将原始文件保存到本地不可变目录，并保留原始文件名与路径。
- **FR-004**：系统必须优先进行 PDF 文本提取，无法获得可用文本时再执行本地 OCR。
- **FR-005**：系统必须把解析结果统一为 `DocumentEvidence`，并记录解析来源、供应商名、供应商版本和错误码。
- **FR-006**：系统必须把 OCR 与文本提取失败的单据标记为 `processing_failed`，且不阻塞同批次其他单据继续执行。

### 5.2 字段抽取与基础校验

- **FR-007**：系统必须抽取购方名称、购方税号、地址、电话、开户银行、银行账户、开票日期、金额、发票号码、销售方名称和发票明细。
- **FR-008**：系统必须为每个抽取字段记录标准化值、置信度和证据片段。
- **FR-009**：系统必须按“字段未出现则忽略、字段出现且高置信则严格比对、字段出现但低置信则进入待复核”的规则执行购方校验。
- **FR-010**：系统必须支持为购方档案维护历史合法别名或旧值，以降低严格匹配的误杀率。
- **FR-011**：系统必须把任一高置信字段不一致的单据判定为 `suggested_reject`。

### 5.3 业务风险分类与重复票检测

- **FR-012**：系统必须使用“业务风险分类”命名该规则层，而不是“业务合规”。
- **FR-013**：系统必须提供有限白名单、黑名单和兜底待复核规则，覆盖 PRD 中列出的典型服务类与实物类场景。
- **FR-014**：系统必须将明细模糊、详见清单缺清单、混合票、OCR 低置信、明细抽取失败等场景统一降级为 `review_required`。
- **FR-015**：系统必须在第一期检查本机历史库内的疑似重复票，最小比对键包含发票号码、发票代码（若有）、金额、日期和销售方名称。
- **FR-016**：系统必须在命中疑似重复时设置 `duplicate_flag = true`，追加 `suspected_duplicate` 风险标记，并将原本建议通过的单据降级为 `review_required`。
- **FR-017**：系统必须保证疑似重复票不进入系统建议通过数量与金额统计。

### 5.4 命名、展示与导出

- **FR-018**：系统必须提供默认命名模板 `YYYYMMDD_金额_发票号码.pdf`。
- **FR-019**：系统必须在金额、日期或发票号码缺失时跳过自动重命名，并保留原因说明。
- **FR-020**：系统必须在文件名冲突时追加 `_2`、`_3` 等后缀。
- **FR-021**：系统必须提供批次结果页，支持按全部、系统建议通过、系统建议驳回、待复核、疑似重复、处理失败筛选。
- **FR-022**：系统必须在批次结果页展示两层金额：批次系统建议通过金额和当前筛选下系统建议通过金额。
- **FR-023**：系统必须在详情抽屉中展示 PDF 预览、识别字段与证据片段、购方字段比对结果、风险分类依据和疑似重复依据。
- **FR-024**：系统必须支持导出系统建议通过 ZIP、问题票 ZIP 和 Excel 台账。

### 5.5 配置、审计与人工复核

- **FR-025**：系统必须提供配置中心，用于维护公司税务档案、业务风险分类规则、命名模板和默认操作者名。
- **FR-026**：系统必须对税务档案、风险规则和命名模板做版本化，并记录变更人、变更时间、变更摘要和变更原因。
- **FR-027**：系统必须在批次创建时固化规则版本快照，确保历史批次可追溯。
- **FR-028**：系统必须支持对待复核或疑似重复票执行人工确认通过、人工确认驳回、保持待复核与备注。
- **FR-029**：系统必须将人工复核写入独立留痕对象，不得直接覆盖 `system_decision`。
- **FR-030**：系统必须采用单进程、单写者的后台状态推进与数据库写入模型。

## 6. 关键实体

- **Batch**：批次主记录，保存处理进度、规则版本快照和系统建议通过汇总。
- **InvoiceRecord**：单张发票的主业务记录，保存文件路径、关键字段、判定状态和显示状态。
- **DocumentEvidence**：统一解析证据模型，屏蔽文本提取与 OCR 供应商差异。
- **ExtractedField**：字段级抽取结果与证据片段。
- **FieldCheck**：购方字段逐项对比结果。
- **RuleVersion**：配置中心中税务档案、风险规则和命名规则的版本快照。
- **ReviewAction**：人工复核留痕。
- **ExportJob**：导出任务与产物记录。

## 7. 非功能要求

- **NFR-001**：系统必须运行在单机本地环境，默认数据保存在本机。
- **NFR-002**：原始文件不可变，重命名文件与导出文件必须写入独立目录。
- **NFR-003**：系统结论必须可解释，至少能回溯到字段证据、校验结果和规则命中原因。
- **NFR-004**：批次处理中断后，系统应能够基于数据库状态和文件系统恢复可见进度。
- **NFR-005**：设计与实现必须保持文档与代码可追溯，遵循 AI-SDLC 的 spec / plan / tasks 真值路径。

## 8. 成功标准

- **SC-001**：标准电子 PDF 发票可在批量导入后建立批次并进入处理流程。
- **SC-002**：系统能对成功解析的发票输出明确的系统建议通过、系统建议驳回或待复核结论。
- **SC-003**：疑似重复票会被独立标记，并从系统建议通过数量与金额统计中排除。
- **SC-004**：结果页能展示批次级和筛选级的系统建议通过金额。
- **SC-005**：详情抽屉能解释字段来源、购方比对结果、风险分类依据和疑似重复依据。
- **SC-006**：配置中心的每次修改都生成可追溯版本，并可被批次快照引用。
- **SC-007**：人工复核动作可追溯，且不会覆盖原始系统判定。
- **SC-008**：导出 ZIP / Excel 的统计口径与界面中的系统建议通过统计一致。
- **SC-009**：整个第一期方案可以在 design 分支完成文档冻结，并在获得 execute 授权后按 tasks 逐批实施。
- **SC-010**：文档阶段所有正式产物均位于 `specs/001-invoice-assistant-mvp/`，而非 `docs/superpowers/*`。
