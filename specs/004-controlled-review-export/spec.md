# 功能规格：发票整理助手四期：受控复核与导出闭环

**工作项编号**：`004-controlled-review-export`  
**创建日期**：2026-04-18  
**状态**：已完成实现与 close-out 文档收口，待后续合并 / 归档决策  
**输入**：把操作者身份从前端自由输入改成后端可信身份；给规则修改、复核、导出挂上角色权限；单票结果补齐基础合规/业务合规/原因/建议动作；明确导出门槛和审计口径。  
**追溯来源**：`发票整理助手_评审终版_重新生成.md`、`specs/001-invoice-assistant-mvp/spec.md`、`specs/003-runtime-state-recovery/spec.md`

## 1. 工作项分类与目标

- **工作项类型**：`governance-hardening`
- **目标**：在现有本地单机产品形态不引入完整登录体系的前提下，收口“谁可以改规则、谁可以复核、谁可以导出、导出前必须满足什么条件、导出后留下什么审计证据”这条财务上线前的最短闭环。
- **核心约束**：
  - 本轮不做 SSO、外部 IAM、审批流或多租户，只做后端可信身份与角色判定。
  - 可信身份必须由后端控制并注入前端展示，前端不再自由填写 `created_by`、`reviewed_by`、`changed_by`。
  - 单票“基础合规 / 业务合规 / 原因 / 建议动作”只基于现有识别、规则和复核结果做结构化输出，不重写判定引擎。
  - 导出门槛必须是可测契约，而不是人工约定；允许与拒绝都要留下审计痕迹。

## 2. 范围

### 2.1 覆盖范围

1. 建立后端可信操作者上下文，并提供前端只读展示入口。
2. 去掉上传、规则修改、复核、导出链路中来自前端自由填写的操作者字段。
3. 为规则修改、复核、导出增加后端角色权限校验与拒绝态响应。
4. 补齐单票结果中的基础合规、业务合规、最终结论、原因与建议动作字段，并同步到导出台账。
5. 定义各导出类型的门槛、拒绝原因与审计口径。
6. 更新四期 work item 文档、任务和执行归档，并用前后端回归锁定行为。

### 2.2 明确不覆盖范围

1. 新增登录页、账户管理、OAuth / SSO 或外部身份提供方集成。
2. 重新设计基础合规或业务合规规则本身，只补充结果表达与治理约束。
3. 新增导出格式、ERP / OA 对接或跨系统审批流。
4. 多主体隔离、多人协作锁定、复核分派队列等更完整后台能力。
5. 对运行时批次恢复、失败重试和活跃批次识别做新的行为扩张；该部分已在三期收口。

## 3. 用户故事与测试

### 用户故事 1：前端只展示后端可信操作者，敏感操作按角色受控（优先级：P0）

作为财务上线责任人，我希望当前操作者身份和角色由后端可信来源提供，前端只能展示不能篡改，这样规则修改、人工复核和结果导出才有可信责任归属。

**优先级说明**：如果操作者身份仍由前端自由输入，任何审计记录都不可信，规则修改、复核结论和导出行为都不能作为财务上线后的责任证据。  
**独立测试**：后端 API 测试验证前端传入的 `created_by` / `reviewed_by` / `changed_by` 不再生效，且缺角色时返回 `403`；前端测试验证界面不再出现可编辑操作者姓名输入框。

#### 场景

1. **Given** 当前后端操作者角色只包含 `reviewer`，**When** 用户调用导出接口，**Then** 接口返回 `403`，不生成导出文件，并写入权限拒绝审计。
2. **Given** 当前后端操作者包含 `config_admin`，**When** 用户创建新规则版本，**Then** 新版本和审计记录中的操作者都来自后端上下文，而不是请求体自由输入。

### 用户故事 2：单票结果具备完整财务解释面（优先级：P0）

作为复核人员，我希望每张票都能直接看到基础合规、业务合规、最终结论、原因和建议动作，这样我不需要从风险标记和零散字段自己拼结论。

**优先级说明**：现状只有 `system_decision`、`risk_flags`、`problem_count` 等技术字段，缺少财务可消费的解释面，复核和导出都没有稳定口径。  
**独立测试**：后端 API / 端到端测试验证单票详情与 Excel 导出台账都返回新增字段，并与当前规则/复核结果保持一致。

#### 场景

1. **Given** 一张基础合规通过但业务规则命中禁止类目的发票，**When** 用户查看单票详情，**Then** 页面返回“基础合规通过、业务合规不通过、最终结论不合规”以及原因和建议动作。
2. **Given** 一张系统判定待复核、后被人工批准的发票，**When** 用户查看详情或导出台账，**Then** 结果同时保留系统原始判定和人工复核后的最终结论。

### 用户故事 3：导出前门槛明确，导出后审计完整（优先级：P0）

作为导出责任人，我希望系统明确哪些批次、哪些票可以导出，并把导出门槛判断和结果都写入审计，这样导出的文件可以作为可追溯财务产物，而不是一次无法解释的文件打包。

**优先级说明**：如果导出接口在处理中、待复核未清或权限不足时仍可执行，财务上线后会直接产生错误归档和责任不清的问题。  
**独立测试**：后端服务 / API 测试验证 `suggested_pass_zip`、`issue_zip`、`excel_manifest` 各自的门槛、成功导出、拒绝导出和审计记录。

#### 场景

1. **Given** 某批次仍有 `review_required` 且未人工处理的发票，**When** 用户导出 `suggested_pass_zip`，**Then** 接口返回 `400`，说明“仍有待复核票据”，并记录导出拒绝审计。
2. **Given** 某批次已经完成处理，且操作者具有 `exporter` 角色，**When** 用户导出 `excel_manifest`，**Then** 系统生成台账并在审计中记录导出类型、结果统计、导出文件路径和操作者角色快照。

## 4. 边界条件

- 后端未配置可信操作者上下文时，受控写操作不得回退到匿名或前端自由输入，应返回明确的配置错误。
- 同一操作者可以拥有多个角色；权限判断按角色集合，而不是单角色字符串。
- 被拒绝的规则修改、复核或导出也要留下审计记录，不能只记录成功事件。
- `issue_zip` 与 `excel_manifest` 可以在批次进入终态后导出，但 `suggested_pass_zip` 不能绕过待复核门槛。
- 补齐单票结果字段时，不得删除现有 `system_decision`、`review_status`、`risk_flags` 等技术字段；要在其上增加财务可读解释层。

## 5. 功能需求

### 5.1 后端可信身份收口

- **FR-001**：系统必须在后端维护当前操作者上下文，至少包含 `actor_id`、`display_name` 与 `roles`，并提供前端只读获取入口。
- **FR-002**：批次创建、规则修改、人工复核和结果导出接口必须从后端操作者上下文写入 `created_by` / `changed_by` / `reviewed_by`，不得再接受前端自由输入同类字段。
- **FR-003**：前端不得继续提供“默认操作者”自由编辑能力，只能展示后端返回的当前操作者和角色。

### 5.2 角色权限控制

- **FR-004**：规则版本创建必须要求 `config_admin` 角色。
- **FR-005**：发票复核动作必须要求 `reviewer` 角色。
- **FR-006**：批次导出必须要求 `exporter` 角色。
- **FR-007**：当角色校验失败时，接口必须返回 `403` 和可读原因，并写入一条权限拒绝审计。

### 5.3 单票结果解释面补齐

- **FR-008**：发票摘要、发票详情和 Excel 导出台账必须补齐 `basic_compliance_status`、`business_compliance_status`、`final_decision`、`decision_reasons` 与 `suggested_actions`。
- **FR-009**：新增结果字段必须优先复用现有规则引擎、重复票判定、失败态和人工复核状态生成，不引入第二套独立判定逻辑。
- **FR-010**：人工复核后的最终结论必须与原始系统判定并存展示，便于审计区分“系统建议”和“人工裁决”。

### 5.4 导出门槛与审计

- **FR-011**：所有导出必须要求批次已进入终态，处理中批次不得导出。
- **FR-012**：`suggested_pass_zip` 仅允许在批次终态且不存在待人工复核票据时导出，且只导出最终结论为可归档通过的票据。
- **FR-013**：`issue_zip` 和 `excel_manifest` 仅要求批次终态，但其导出结果必须明确反映当前待复核、处理失败和不合规票据状态。
- **FR-014**：每次导出尝试都必须写入审计，至少记录操作者身份、角色快照、导出类型、门槛判定结果、摘要统计、输出路径（成功时）和拒绝原因（失败时）。

### 5.5 验证与追溯

- **FR-015**：系统必须补齐后端回归测试，覆盖可信身份派生、角色权限、导出门槛和审计落账。
- **FR-016**：系统必须补齐前端回归测试，覆盖当前操作者只读展示与自由输入移除。
- **FR-017**：四期 work item 文档、任务和执行日志必须与本次实现保持可追溯，不保留脚手架占位文本。

## 6. 关键实体

- **TrustedActor**：后端可信操作者上下文，提供身份、显示名和角色集合。
- **ActorRole**：角色集合，至少包含 `config_admin`、`reviewer`、`exporter`。
- **InvoiceComplianceSummary**：单票财务解释层，负责输出基础合规、业务合规、最终结论、原因和建议动作。
- **ExportGateEvaluation**：导出门槛判断结果，负责说明是否允许导出及原因。
- **AuditLog**：审计记录实体，负责保存成功或拒绝的规则修改、复核和导出行为。

## 7. 非功能要求

- **NFR-001**：四期实现必须保持当前 FastAPI + SQLAlchemy + SQLite + 本地文件系统 + React 架构，不引入外部鉴权基础设施。
- **NFR-002**：权限判断必须发生在后端，前端按钮禁用仅作为体验补充，不能替代后端校验。
- **NFR-003**：新增单票结果字段必须结构化返回，避免再次退回为仅靠拼接文案或手工判断。
- **NFR-004**：导出台账字段与单票详情字段必须共用同一套解释口径，避免 UI 与导出各说各话。
- **NFR-005**：四期验证优先通过现有 pytest / vitest 回归与执行归档完成，不以口头说明代替证据。

## 8. 成功标准

- **SC-001**：界面不再出现前端可编辑的默认操作者输入，且上传、复核、规则修改、导出所记录的操作者都来自后端可信上下文。
- **SC-002**：缺少 `config_admin`、`reviewer`、`exporter` 角色时，对应受控接口稳定返回 `403`，并能查到拒绝审计记录。
- **SC-003**：单票详情和 Excel 导出台账都能展示基础合规、业务合规、最终结论、原因和建议动作。
- **SC-004**：当批次存在待复核票据时，`suggested_pass_zip` 会被稳定拒绝；当门槛满足时，导出成功并留下完整审计。
- **SC-005**：`spec.md`、`plan.md`、`tasks.md` 与执行日志可回溯本次财务治理收口范围，不保留脚手架占位文本。
